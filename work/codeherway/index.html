CodeHerWay Education Platform

Hero Summary

CodeHerWay is a product-minded learning platform, not just a portfolio landing page. It combines structured frontend curriculum, authenticated learner state, quiz feedback, XP and streak mechanics, progress recovery, and documentation that makes tradeoffs visible to reviewers.

The goal was to prove that I can think across product experience and implementation details: how a beginner moves through lessons, how the app earns trust when saving progress, and how the codebase can keep growing without turning every new lesson into one-off UI.

Product Problem

Beginner learners often face too many choices at once: scattered resources, unclear next steps, inconsistent feedback, and progress that disappears between sessions. For women entering tech, that friction can make a difficult subject feel more isolating than it needs to be.

CodeHerWay addresses that by giving learners a clear path, small completion loops, visible progress, and a product tone that supports confidence while still respecting the rigor of learning HTML, CSS, JavaScript, and React.

Honest boundary: I do not claim user growth, retention, revenue, or learner outcome metrics for this project. The case study focuses on the product and engineering decisions I can directly prove.

Target Users

  • Women entering tech who want a structured, less intimidating path into frontend development.
  • Self-directed learners who benefit from visible progress, immediate feedback, and a clear next lesson.
  • Returning learners who need their progress, streak, XP, bookmarks, and last position to feel dependable.
  • Hiring managers reviewing whether the project behaves like a real product with real state, constraints, and quality gates.

My Role

I owned the product end to end: frontend architecture, learning experience, quiz and lesson UI, reward logic, progress persistence, Supabase integration, accessibility improvements, test coverage, documentation, and release-readiness planning.

This meant working at both levels: shaping the learner journey and making the underlying system resilient enough that progress, rewards, and lesson state were not just decorative UI.

Product Goals

  • Make the first learning session obvious: sign in, choose a course, begin a lesson, answer a quiz, and see progress update.
  • Keep the learner oriented with course structure, lesson status, current position, review surfaces, and completion feedback.
  • Use XP and streaks to reward consistency over volume so the platform encourages steady practice instead of grind behavior.
  • Build the product with enough real infrastructure that recruiters can evaluate engineering judgment, not just static visuals.

System overview

From sign-in to reward state

The core loop connects learner intent, lesson activity, quiz completion, persistence, and reward updates without making the interface feel heavy.

  1. 01

    User signs in

    Supabase auth establishes the learner session and separates guest exploration from saved progress.

  2. 02

    User views lessons

    The lesson tree presents beginner-friendly modules with a visible current position and next step.

  3. 03

    User completes quizzes

    Quiz feedback gives learners a fast read on what worked, what needs review, and what to revisit.

  4. 04

    Progress is persisted

    Lesson completion, bookmarks, last position, and review state are written through the progress layer.

  5. 05

    XP and streak update

    Reward state responds to consistent practice while keeping motivational progress distinct from mastery.

Core User Flow

  1. A learner signs in or starts from the public learning entry point.
  2. They choose a course path and land in a lesson view with navigation, context, and supporting tools.
  3. They read structured lesson content, complete tasks or quizzes, and receive immediate feedback.
  4. The app saves completion and position state so the learner can return without rebuilding context.
  5. XP, streak, and progress surfaces update to make steady practice visible without overstating the result.

UX Decisions

Guided learning flow

Lessons are arranged as a path rather than a loose resource library, which helps beginners understand what to do next.

Immediate feedback

Quiz states and review copy respond quickly, so mistakes become part of the learning loop instead of dead ends.

Visible progress

Completion, last position, bookmarks, XP, and streak state make return sessions feel continuous.

Reduced overwhelm

The interface favors clear next actions, compact supporting panels, and calm language over dense dashboards.

Consistency over volume

Reward language is designed to encourage regular practice, not marathon sessions or inflated mastery claims.

Engineering Decisions

Supabase auth

Authentication gives the product a real learner boundary and supports saved state beyond a static demo.

Persisted progress

The progress layer handles lesson completion, last position, bookmarks, sync status, and recovery messaging.

Beginner-friendly lessons

Lesson content is structured so concepts, examples, checks, and review states can be presented consistently.

Async UI states

Loading, saving, error, recovery, and offline-adjacent states are treated as part of the product experience.

Scalable lesson structure

Courses, modules, quizzes, and supporting panels are separated enough to add lessons without redesigning the app each time.

Decision Receipts

  • Constraint

    Beginner motivation needs to feel encouraging without implying verified mastery.

  • Decision

    XP and streaks reward consistency, while quiz review remains the place for learning feedback.

  • Tradeoff

    The reward loop supports momentum, but retention and completion impact still need real learner validation.

Accessibility Considerations

  • Keyboard navigation and focus handling matter across lessons, route boundaries, panels, dialogs, and mobile tool surfaces.
  • Progress and quiz feedback use clear status copy so learners are not forced to infer meaning from color or motion alone.
  • The learning interface keeps motion restrained and purposeful, with reduced-motion support treated as a product requirement.
  • Semantic structure, headings, readable contrast, and touch-friendly controls support both screen reader and mobile use.

Performance Considerations

  • Vite keeps local iteration fast while the product remains small enough for focused performance budgets.
  • Route-level and lesson-level concerns are documented so future optimization can target real learner surfaces.
  • Performance work is framed around measurable checks, bundle awareness, and mobile learning flow rather than visual flash.
  • The next phase should keep adding Lighthouse evidence, bundle policy checks, and screenshot-based regression review.

Current Status

CodeHerWay is an active portfolio product with real frontend architecture, authenticated-state planning, progress and reward systems, tests, documentation, and deployment paths. It is credible as a product-engineering case study, but I present it honestly as still moving through staging validation and production-readiness hardening.

What is proven here: product thinking, frontend architecture, state design, accessibility intent, testing discipline, and the ability to document tradeoffs clearly.

What I Learned

The strongest education-product evidence comes from connecting the learner-facing interface to real state rules: auth, progress, quiz review, and recovery paths. The next level is validating which of those choices actually helps beginners continue.

What I Would Improve Next

  • Validate the authenticated learner journey against a configured staging Supabase project.
  • Add the real dashboard screenshot and keep it current with the deployed product.
  • Deepen Lighthouse reporting, bundle budgets, and mobile regression evidence.
  • Instrument learning friction points such as quiz retries, drop-off, and lesson resume behavior before making outcome claims.
  • Continue reducing global CSS and clarifying reusable lesson, panel, and progress primitives.
